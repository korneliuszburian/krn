# Goal 037: KRN Engineering Kernel Reset

## Status

Active child goal for `docs/goals/goal-006.md`.

Starts after `docs/goals/goal-036.md`, but changes the operating direction. `goal-036` may remain a helper for bounded research, not the default next product milestone.

## Objective

Reset KRN work away from artifact/eval churn and toward senior-engineering product construction.

KRN must be built as a real Codex operating layer: compact quality language, multi-layer memory behavior, source/practice mechanisms, smallest production-shaped code slices, continuous review, and measured feedback.

## Mechanism

Use the KRN engineering kernel as the default steering language:

```text
mechanism-first
  -> bottleneck-led
  -> production-shaped
  -> context-budgeted
  -> diff-literate
  -> review-minimizing
  -> memory-operative
  -> proof-carrying
  -> deletion-friendly
```

This is the compressed instruction set that should make future Codex work behave more like a senior engineer and less like an artifact generator.

## Boundaries

- Do not treat `docs/memory/**` as the final memory system.
- Do not add a new eval module unless it protects a durable contract, known regression, or measured hypothesis.
- Do not add a new typed object unless another product surface consumes it or it removes real review burden.
- Do not use `KrnResearchPack` as proof of research quality until a real worker fills it and a review path consumes it.
- Do not make ChatGPT bridge the product core; it remains a second-opinion/control-quality layer.
- Do not claim breakthrough until a measurable repeated failure mode improves.

## Acceptance Evidence

- `AGENTS.md` contains the compact engineering kernel.
- `docs/memory/product/2026-06-20--krn-senior-engineering-lens.md` explains the kernel and names what actually raises quality.
- `docs/product/final-product-plan.md` describes memory as multi-layer operational behavior, not a docs folder.
- `docs/goals/goal-006.md` says artifact/eval churn is not the default operating mode.
- The next implementation slice is selected by bottleneck, not by "what artifact can be added next."

## Next Implementation Slice

First candidate: **memory selection/application proof**.

Reason: current file-backed pattern notes do not reliably change agent behavior. KRN needs one small path that:

```text
task intent
  -> select minimal relevant patterns/goals/sources
  -> explain why selected
  -> apply them to a concrete action or review decision
  -> record whether they reduced the target failure mode
```

The implementation should be smaller than a new framework. A CLI/report path or directly consumed skill contract is acceptable if it proves selection and application. A passive doc is not enough.

## Disproves Completion

- The next work adds another artifact because it is easy, not because it fixes a product bottleneck.
- Memory remains a file bank with no selection/application proof.
- The kernel becomes decorative language that does not change planning, code, review, or deletion decisions.
- A green eval is presented as quality improvement without showing the behavior it protects.
